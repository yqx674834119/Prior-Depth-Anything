# 2026-03-23 示例数据到线上接口定义

## 结论

这个项目本质是：

- ESP32 发送 `RGB 图像`
- ESP32 发送 `ToF 深度数据`
- Android 根据**预先标定好的 ToF-相机映射关系**完成融合

也就是说，后期实时处理应放在 Android 端，而不是 ESP32 端。

## 示例数据说明

`resource/DataFusion/demo/demo/room/1647156045.486036.h5` 在 [`batch_run.py`](/C:/Users/Shawn/StudioProjects/Prior-Depth-Anything/batch_run.py) 中读取了：

- `rgb`
- `fr`
- `hist_data`
- `mask`
- `depth`

其中：

- `rgb`
  - 相机图像
- `hist_data`
  - ToF 每个 zone 的深度分布数据
  - 当前脚本实际使用 `hist_data[:, 0]` 作为每个 zone 的代表深度
- `mask`
  - 每个 ToF zone 是否有效
- `fr`
  - ToF zone 投影到相机图像中的区域
  - 这是标定结果，不是每帧都要重新算
- `depth`
  - 真实深度图，只用于离线评估

`sparse_prior_8x8.npy` 的生成依赖：

- `fr + hist_data + mask`

含义是：

- 用 `mask` 过滤无效 zone
- 用 `hist_data[:, 0]` 取每个 zone 的深度值
- 用 `fr` 把这个深度值放到相机图像对应区域

## 线上接口建议

### ESP32 负责发送

#### 1. 图像

- `GET http://192.168.4.1:81/`
- 返回当前 JPEG 图像

#### 2. ToF 深度

推荐最小包：

```json
{
  "d": [64个深度值],
  "mask": [64个有效位]
}
```

其中：

- `d` 对应 `hist_data[:, 0]`
- `mask` 对应 zone 是否有效

### Android 负责处理

Android 本地保存一份**固定标定表**：

- `fr = [[sy, sx, ey, ex], ... 共64组]`

然后每帧执行：

1. 接收当前 RGB 图像
2. 接收 `d + mask`
3. 用本地 `fr` 映射表重建稀疏先验
4. 输入模型做 ToF 与相机融合

## 最终最小实时输入

实时系统真正需要的只有：

1. `RGB 图像`
2. `64 个 ToF 深度值`
3. `64 个有效位`
4. 一份预先标定好的固定 `fr` 映射表

## 一句话版本

ESP32 发图像和 ToF 深度，Android 用本地保存的标定映射关系把 ToF 投影到图像上，再完成融合。
